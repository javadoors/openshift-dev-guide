# 使用kubeadm安装业务集群的完整流程与依赖关系
## 一、整体流程概览
kubeadm安装业务集群分为**准备阶段**、**控制平面部署**、**工作节点扩展**和**集群验证**四个主要阶段，每个阶段都有明确的依赖关系和执行顺序。
```
┌─────────────────────────────────────────────────────────────────────┐
│                        完整安装流程                                 │
└─────────────────────────────────────────────────────────────────────┘

1. 准备阶段
   └──> 2. 控制平面部署
          └──> 3. 工作节点扩展
                 └──> 4. 集群验证
```
## 二、详细流程与依赖关系
### 2.1 准备阶段
#### 2.1.1 硬件与操作系统准备
| 步骤 | 操作内容 | 依赖 | 详细说明 |
|------|---------|------|----------|
| 1.1.1 | 硬件准备 | - | 控制平面：2C4G+，工作节点：1C2G+ |
| 1.1.2 | 操作系统安装 | - | 支持的Linux发行版（Ubuntu 20.04+/CentOS 7.8+/RHEL 7.8+） |
| 1.1.3 | 网络配置 | - | 所有节点间网络互通，关闭防火墙/selinux，设置静态IP |
| 1.1.4 | 主机名设置 | - | 每个节点设置唯一主机名，配置/etc/hosts解析 |

**依赖关系**：无前置依赖，为所有后续步骤提供基础环境。
#### 2.1.2 系统依赖安装
| 步骤 | 操作内容 | 依赖 | 详细说明 |
|------|---------|------|----------|
| 1.2.1 | 安装基础工具 | 操作系统 | `apt-get install -y curl apt-transport-https gnupg2` 或 `yum install -y curl yum-utils` |
| 1.2.2 | 配置时间同步 | 基础工具 | 安装并配置chrony或ntpd，确保所有节点时间同步 |
| 1.2.3 | 禁用交换分区 | 操作系统 | `swapoff -a` 并修改/etc/fstab永久禁用 |
| 1.2.4 | 加载内核模块 | 操作系统 | 加载br_netfilter、overlay等Kubernetes所需内核模块 |
| 1.2.5 | 配置内核参数 | 内核模块 | 设置net.bridge.bridge-nf-call-iptables等参数 |

**依赖关系**：依赖操作系统安装完成，为容器运行时和Kubernetes组件提供必要的系统环境。
#### 2.1.3 容器运行时安装
| 步骤 | 操作内容 | 依赖 | 详细说明 |
|------|---------|------|----------|
| 1.3.1 | 安装containerd | 系统依赖 | 安装containerd.io包，版本需与Kubernetes版本兼容 |
| 1.3.2 | 配置containerd | containerd安装 | 生成默认配置：`containerd config default > /etc/containerd/config.toml` |
| 1.3.3 | 启用systemd cgroup | containerd配置 | 修改配置文件，设置`SystemdCgroup = true` |
| 1.3.4 | 启动并启用containerd | containerd配置 | `systemctl enable --now containerd` |

**依赖关系**：依赖系统依赖安装完成，为Kubernetes组件提供容器运行环境。
#### 2.1.4 Kubernetes组件安装
| 步骤 | 操作内容 | 依赖 | 详细说明 |
|------|---------|------|----------|
| 1.4.1 | 添加Kubernetes源 | 基础工具 | 添加Kubernetes官方apt/yum源 |
| 1.4.2 | 安装kubeadm/kubelet/kubectl | Kubernetes源 | 安装指定版本：`apt-get install -y kubeadm=1.28.2-00 kubelet=1.28.2-00 kubectl=1.28.2-00` |
| 1.4.3 | 锁定Kubernetes版本 | Kubernetes组件安装 | 防止自动升级：`apt-mark hold kubeadm kubelet kubectl` |
| 1.4.4 | 配置kubelet | Kubernetes组件安装 | 创建/etc/default/kubelet文件，配置cgroup驱动等参数 |
| 1.4.5 | 启动并启用kubelet | kubelet配置 | `systemctl enable --now kubelet` |

**依赖关系**：依赖容器运行时安装完成，为集群部署提供核心组件。
### 2.2 控制平面部署
#### 2.2.1 初始化第一个控制平面节点
| 步骤 | 操作内容 | 依赖 | 详细说明 |
|------|---------|------|----------|
| 2.2.1 | 初始化控制平面 | Kubernetes组件 | `kubeadm init --control-plane-endpoint "cluster-endpoint:6443" --pod-network-cidr 192.168.0.0/16 --kubernetes-version v1.28.2` |
| 2.2.2 | 配置kubectl | 控制平面初始化 | 复制kubeconfig文件：`mkdir -p $HOME/.kube && cp -i /etc/kubernetes/admin.conf $HOME/.kube/config` |
| 2.2.3 | 安装网络插件 | 控制平面初始化 | 安装Calico：`kubectl apply -f https://docs.projectcalico.org/manifests/calico.yaml` |
| 2.2.4 | 验证控制平面状态 | 网络插件安装 | 检查节点和Pod状态：`kubectl get nodes && kubectl get pods -n kube-system` |

**依赖关系**：
- 依赖准备阶段所有步骤完成
- 网络插件安装依赖控制平面初始化完成
- 控制平面状态验证依赖网络插件安装完成
#### 2.2.2 扩展控制平面节点（高可用）
| 步骤 | 操作内容 | 依赖 | 详细说明 |
|------|---------|------|----------|
| 2.3.1 | 获取控制平面加入令牌 | 第一个控制平面 | 在第一个控制平面执行：`kubeadm token create --print-join-command` |
| 2.3.2 | 复制证书密钥 | 第一个控制平面 | 复制证书密钥到新控制平面节点 |
| 2.3.3 | 加入新控制平面节点 | 令牌和证书 | 在新控制平面执行：`kubeadm join cluster-endpoint:6443 --token <token> --discovery-token-ca-cert-hash <hash> --control-plane` |
| 2.3.4 | 验证控制平面集群 | 新控制平面加入 | 检查所有控制平面节点状态：`kubectl get nodes` |

**依赖关系**：
- 依赖第一个控制平面部署完成
- 新控制平面加入依赖令牌和证书准备完成

### 2.3 工作节点扩展

#### 2.3.1 加入工作节点

| 步骤 | 操作内容 | 依赖 | 详细说明 |
|------|---------|------|----------|
| 3.1.1 | 获取工作节点加入令牌 | 控制平面 | 在控制平面执行：`kubeadm token create --print-join-command` |
| 3.1.2 | 加入工作节点 | 令牌 | 在工作节点执行：`kubeadm join cluster-endpoint:6443 --token <token> --discovery-token-ca-cert-hash <hash>` |
| 3.1.3 | 验证工作节点状态 | 工作节点加入 | 检查所有节点状态：`kubectl get nodes` |

**依赖关系**：
- 依赖控制平面部署完成
- 工作节点加入依赖令牌准备完成
### 2.4 集群验证
#### 2.4.1 功能验证
| 步骤 | 操作内容 | 依赖 | 详细说明 |
|------|---------|------|----------|
| 4.1.1 | 创建测试Pod | 集群正常运行 | `kubectl run test-pod --image=nginx` |
| 4.1.2 | 验证Pod网络 | 测试Pod创建 | 检查Pod是否能正常访问网络：`kubectl exec -it test-pod -- ping -c 3 8.8.8.8` |
| 4.1.3 | 验证服务发现 | Pod网络正常 | 创建Service并验证访问：`kubectl expose pod test-pod --port=80 --type=ClusterIP` |

**依赖关系**：
- 依赖工作节点扩展完成
- 功能验证依赖集群正常运行

## 四、关键组件版本兼容性
| Kubernetes版本 | 推荐containerd版本 | 推荐CNI版本 | 支持的Linux发行版 |
|---------------|-------------------|-------------|------------------|
| v1.28.x       | 1.7.7-1.7.11      | Calico 3.26+ | Ubuntu 20.04+/CentOS 7.9+/RHEL 7.9+ |
| v1.27.x       | 1.7.6-1.7.10      | Calico 3.25+ | Ubuntu 20.04+/CentOS 7.9+/RHEL 7.9+ |
| v1.26.x       | 1.6.24-1.7.9      | Calico 3.24+ | Ubuntu 20.04+/CentOS 7.9+/RHEL 7.9+ |

**注意**：
- kubeadm、kubelet、kubectl必须使用相同版本
- 容器运行时版本需与Kubernetes版本兼容
- CNI插件版本需与Kubernetes版本兼容
## 五、常见依赖问题与解决方案
### 5.1 容器运行时问题
| 问题 | 原因 | 解决方案 |
|------|------|----------|
| containerd无法启动 | systemd cgroup未配置 | 修改/etc/containerd/config.toml，设置`SystemdCgroup = true` |
| 镜像拉取失败 | 网络问题或镜像源 | 配置containerd镜像加速：`registry.mirrors` |
### 5.2 Kubernetes组件问题
| 问题 | 原因 | 解决方案 |
|------|------|----------|
| kubelet无法启动 | cgroup驱动不匹配 | 确保kubelet和containerd使用相同的cgroup驱动 |
| 控制平面初始化失败 | 端口被占用 | 检查6443、2379、2380等端口是否被占用 |
### 5.3 网络问题
| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 节点NotReady | 网络插件未安装 | 安装兼容的CNI插件 |
| Pod间无法通信 | CNI配置错误 | 检查CNI配置和网络策略 |
## 六、最佳实践
1. **版本管理**：
   - 使用固定版本安装，避免自动升级
   - 定期检查版本兼容性，提前规划升级
2. **高可用设计**：
   - 控制平面节点数量为奇数（3/5个）
   - 使用外部etcd或堆叠etcd（根据规模选择）
   - 配置负载均衡器作为控制平面endpoint
3. **网络优化**：
   - 选择性能良好的CNI插件（如Calico、Cilium）
   - 合理规划Pod CIDR和Service CIDR
   - 配置网络策略增强安全性
4. **存储配置**：
   - 为etcd配置高性能存储
   - 为工作节点配置适当的存储类
   - 考虑使用CSI插件管理存储
5. **监控与日志**：
   - 部署Prometheus/Grafana监控集群
   - 配置ELK或Loki收集日志
   - 设置告警规则监控集群健康状态
## 七、总结
使用kubeadm安装业务集群是一个**依赖链清晰**的流程，从基础环境准备到集群功能验证，每个步骤都有明确的前置依赖。核心依赖关系为：
```
硬件/操作系统 → 系统依赖 → 容器运行时 → Kubernetes组件 → 控制平面初始化 → 网络插件 → 工作节点 → 集群验证
```
遵循正确的依赖顺序和版本兼容性要求，能够确保集群安装过程顺利完成，并为后续的集群管理和扩展打下坚实基础。
